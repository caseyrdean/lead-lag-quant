# Phase 05.1 Plan 03: broadcast_sync Thread-Safety Fix Summary

**One-liner:** Replaced `loop.create_task()` with `asyncio.run_coroutine_threadsafe()` in `broadcast_sync()` to eliminate silent WebSocket message loss from background threads.

## What Was Built

### BUGFIX-08: Thread-safe broadcast scheduling

`broadcast_sync()` in `api/ws.py` previously used `loop.create_task()` which is **not thread-safe**. When `BackgroundPricePoller` or `PipelineScheduler` calls `broadcast_sync()` from a worker thread while the event loop is running in the main thread, `create_task()` can silently drop messages or corrupt loop state.

`asyncio.run_coroutine_threadsafe()` is the stdlib-correct API: it uses a thread-safe queue to submit coroutines to a running event loop from another thread, and returns a `concurrent.futures.Future` (discarded here for fire-and-forget behavior).

**Before:**
```python
loop.create_task(self.broadcast(event_type, data))  # thread-unsafe
```

**After:**
```python
asyncio.run_coroutine_threadsafe(self.broadcast(event_type, data), loop)  # thread-safe
```
