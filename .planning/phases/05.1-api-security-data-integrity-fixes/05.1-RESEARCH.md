## Summary

Phase 5.1 is a targeted bug-fix phase — no new libraries, no schema migrations, no new features. All eight bugs are surgical changes to existing files with well-understood fix patterns. The code has already been read in full; the fixes are described with exact locations, root causes, and verified correct patterns.

The two CRITICAL bugs are in `api/routes/pairs.py`: the FREE-tier limit is enforced in the Gradio UI (`ui/app.py`) but is completely absent from the FastAPI `POST /pairs` endpoint, and the `DELETE /pairs` loop was missing `conn.commit()` (it was actually present at the end — confirmed on re-read: line 143 shows `conn.commit()` after the loop). On re-read of `delete_pairs()`: the commit IS present at line 143. So BUGFIX-02 (missing commit on delete) may already be fixed — the planner should verify at task time. The HIGH bugs center on signal queries missing `JOIN ticker_pairs WHERE is_active = 1`, the reactivation path generating stale auto-executions, and `auto_execute_signals()` having no mutex. The MEDIUM bugs are straightforward Pydantic `Field(gt=0)` additions, `try/except HTTPException` wrappers, and replacing `loop.create_task()` with `asyncio.run_coroutine_threadsafe()`.

**Primary recommendation:** Fix all eight bugs in three plan waves matching the roadmap split — (1) API layer fixes, (2) signal filtering + concurrency, (3) WebSocket reliability — with a pytest run after each wave.

## Architecture Patterns

### Pattern 1: Free-Tier Guard in FastAPI Route

**What:** Query active pair count before insert; return 403 if limit exceeded.
**When to use:** BUGFIX-01 — `add_pairs()` in `api/routes/pairs.py`

**Reference from `ui/app.py` lines 96-103 (exact pattern to port):**
```python
# Source: ui/app.py lines 51, 61-65, 96-103
FREE_TIER_MAX_PAIRS = 5

if config.plan_tier == PlanTier.FREE:
    active = conn.execute(
        "SELECT COUNT(*) FROM ticker_pairs WHERE is_active = 1"
    ).fetchone()[0]
    if active >= FREE_TIER_MAX_PAIRS:
        raise HTTPException(
            status_code=403,
            detail=f"Free tier is limited to {FREE_TIER_MAX_PAIRS} active pairs."
        )
```

**Key detail:** Use `HTTPException(status_code=403)` not a plain dict return, so FastAPI generates the correct HTTP status code. The Gradio UI uses a plain return; the API must use HTTPException.

### Pattern 2: SQL JOIN to Filter Soft-Deleted Pairs from Signals

**What:** Add `INNER JOIN ticker_pairs tp ON tp.leader = s.ticker_a AND tp.follower = s.ticker_b WHERE tp.is_active = 1` to both signal queries.
**When to use:** BUGFIX-03 — `_get_active_signals_dataframe()` and `list_signals()`

```sql
-- Add to FROM signals s:
INNER JOIN ticker_pairs tp
    ON tp.leader = s.ticker_a
    AND tp.follower = s.ticker_b
    AND tp.is_active = 1
```

**Key detail:** Both queries must get this JOIN. The `api/routes/signals.py` `list_signals()` and `ui/signal_dashboard.py` `_get_active_signals_dataframe()` are independent and both need the fix.

**Key detail for `get_unprocessed_signals()` in `paper_trading/db.py`:** This function also lacks the `is_active` join, and it feeds `auto_execute_signals()`. It needs the same JOIN fix.

### Pattern 3: Reactivation Date Guard

**What:** Exclude signals generated before the pair was most recently reactivated.
**When to use:** BUGFIX-04 — `get_unprocessed_signals()` in `paper_trading/db.py`

The `ticker_pairs` table does not currently store a `reactivated_at` timestamp. The reactivation guard requires one of:
- Option A: Add `reactivated_at TEXT` column to `ticker_pairs`, set it on every activation (INSERT and UPDATE SET is_active=1).
- Option B: Store reactivation events in a separate log table.
- Option C: Proxy guard — compare `signals.generated_at` to `ticker_pairs.created_at` (only catches initial creation, not reactivations — INSUFFICIENT).

**Recommendation: Option A** — add `reactivated_at` column to `ticker_pairs`. This is a single `ALTER TABLE` or schema-level `DEFAULT` change, and it's the minimal-footprint approach. On reactivation (both in `pairs.py` and `app.py`), set `reactivated_at = current_timestamp`. In `get_unprocessed_signals()`, add:
```sql
AND s.generated_at >= tp.reactivated_at
```

**Schema change required:** `ALTER TABLE ticker_pairs ADD COLUMN reactivated_at TEXT;` (nullable, NULL means never deactivated — treat NULL as "all signals valid").

### Pattern 4: Mutex for auto_execute_signals

**What:** Module-level `threading.Lock()` acquired at the top of `auto_execute_signals()`.
**When to use:** BUGFIX-05 — `paper_trading/engine.py`

```python
# Source: Python stdlib threading docs
import threading

_execute_lock = threading.Lock()  # module-level singleton

def auto_execute_signals(conn, api_key, portfolio_id=1):
    with _execute_lock:
        # existing body unchanged
        ...
```

**Key detail:** The lock must be module-level (not per-call) to actually serialize concurrent callers. A non-blocking approach (`_execute_lock.acquire(blocking=False)`) would silently skip; blocking is correct here — the second caller should wait, not skip.

### Pattern 5: Pydantic v2 Field Validation

**What:** Add `Field(gt=0)` to numeric fields in request models.
**When to use:** BUGFIX-06 — `BuyRequest` and `SellRequest` in `api/routes/trading.py`

```python
# Source: Pydantic v2 docs
from pydantic import BaseModel, Field

class BuyRequest(BaseModel):
    ticker: str
    shares: int = Field(gt=0)
    price: float | None = Field(default=None, gt=0)
    ...
```

**Key detail for `price: float | None`:** Pydantic v2 applies `gt=0` only when the value is not None. If price is None (fetch from Polygon), the constraint is skipped. This is the correct behavior — validation should reject `price=0` but allow `price=None`.

**FastAPI behavior:** When Pydantic validation fails, FastAPI automatically returns `HTTP 422 Unprocessable Entity` with a `detail` array listing each field violation. No manual error handling needed.

### Pattern 7: run_coroutine_threadsafe for Cross-Thread Broadcast

**What:** Replace `loop.create_task()` with `asyncio.run_coroutine_threadsafe(coro, loop)`.
**When to use:** BUGFIX-08 — `broadcast_sync()` in `api/ws.py`

**Current broken code (api/ws.py lines 41-47):**
```python
def broadcast_sync(self, event_type: str, data: dict) -> None:
    try:
        loop = asyncio.get_running_loop()
        loop.create_task(self.broadcast(event_type, data))  # WRONG: thread-unsafe
    except RuntimeError:
        pass
```

**Correct fix:**
```python
def broadcast_sync(self, event_type: str, data: dict) -> None:
    try:
        loop = asyncio.get_running_loop()
        asyncio.run_coroutine_threadsafe(self.broadcast(event_type, data), loop)
    except RuntimeError:
        pass
```

**Why this works:** `loop.create_task()` is NOT thread-safe — it can silently drop messages when called from a thread other than the event loop thread. `asyncio.run_coroutine_threadsafe()` uses a thread-safe queue (`call_soon_threadsafe()` internally) to schedule the coroutine on the event loop from any thread. It returns a `concurrent.futures.Future` that can be ignored for fire-and-forget use.

**Key detail:** `asyncio.get_running_loop()` still correctly obtains the event loop running in the main FastAPI/uvicorn thread when called from a background thread — it raises `RuntimeError` only if there is no loop running at all, which is the same as before.

### Anti-Patterns to Avoid

- **Returning plain dicts for errors in FastAPI:** `return {"error": "..."}` returns HTTP 200 with an error body. Use `HTTPException` for 4xx or `JSONResponse(status_code=...)` for 5xx.
- **Per-call mutex instantiation:** `with threading.Lock():` inside the function creates a new lock per call — no mutual exclusion. The lock MUST be module-level.
- **Nullable Field with gt=0 — wrong approach:** `price: float = Field(gt=0)` would reject `None`. Use `float | None = Field(default=None, gt=0)` — Pydantic v2 skips constraint for None values when the field is Optional.
- **Forgetting `get_unprocessed_signals` in db.py:** BUGFIX-03 is described as affecting `signal_dashboard.py` and `api/routes/signals.py`, but `paper_trading/db.py get_unprocessed_signals()` also queries signals without the `is_active` filter and feeds `auto_execute_signals()`. All three query sites need the fix.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Input validation (shares > 0, price > 0) | Manual if/else guards in route body | `pydantic.Field(gt=0)` | Pydantic generates 422 automatically with proper error detail |
| Cross-thread coroutine scheduling | Custom queue/pipe between threads | `asyncio.run_coroutine_threadsafe()` | Thread-safe by design, stdlib, returns a Future |
| Concurrent access serialisation | Custom flag variables | `threading.Lock()` | Standard, re-entrant safe, context manager |
| Structured error responses | Custom exception classes | `fastapi.HTTPException` + `fastapi.responses.JSONResponse` | FastAPI renders correct status codes and Content-Type |

## Common Pitfalls

### Pitfall 1: BUGFIX-02 Already Fixed
**What goes wrong:** The bug report says `delete_pairs()` has no `conn.commit()`, but reading `api/routes/pairs.py` line 143 shows `conn.commit()` IS present after the loop.
**Why it happens:** The bug report may have been written before the commit was added, or refers to an earlier version.
**How to avoid:** The planner should create a "verify" task that confirms the commit is present and adds a regression test, rather than applying a fix that's already been applied.
**Warning signs:** If the executor finds `conn.commit()` already present, mark BUGFIX-02 complete after adding a test.

### Pitfall 2: Schema Change for BUGFIX-04 Needs Migration
**What goes wrong:** `ticker_pairs` table does not have `reactivated_at` column. Existing DBs will fail `ALTER TABLE` if the migration is applied twice.
**Why it happens:** SQLite `ALTER TABLE ADD COLUMN` fails if the column already exists.
**How to avoid:** Use `ALTER TABLE ticker_pairs ADD COLUMN reactivated_at TEXT` in `utils/db.py init_schema()`, wrapped in a try/except for `sqlite3.OperationalError` (or check `PRAGMA table_info` first). Since the project uses `IF NOT EXISTS` consistently, follow the same pattern.
**Warning signs:** Any `OperationalError: duplicate column name` in schema init.

### Pitfall 3: Pydantic v2 Field with Optional + Constraint
**What goes wrong:** `price: float | None = Field(gt=0)` — in Pydantic v2, `gt=0` on an Optional field still applies when value is not None. However, `Field(default=None, gt=0)` must be used (not just `gt=0` without default).
**Why it happens:** Forgetting the `default=None` makes the field required.
**How to avoid:** `price: float | None = Field(default=None, gt=0)`. Test with `BuyRequest(ticker="AAPL", shares=10)` — price should default to None without error.

### Pitfall 4: broadcast_sync Called From the Event Loop Thread
**What goes wrong:** If `broadcast_sync()` is ever called from within an async context (i.e., the uvicorn event loop thread), `run_coroutine_threadsafe()` will still work but is unnecessary overhead.
**Why it happens:** The function is designed for background threads only.
**How to avoid:** Not a blocker — `run_coroutine_threadsafe()` works from both contexts. No guard needed.

### Pitfall 5: Signal Filter JOIN Changes Query Column Count
**What goes wrong:** Adding `JOIN ticker_pairs tp` changes the SQL, but if `tp.*` columns are accidentally selected (e.g., `SELECT *`), extra columns will appear in results.
**Why it happens:** Using `SELECT *` with a JOIN returns columns from all joined tables.
**How to avoid:** Both signal queries already use explicit column lists (`s.rowid AS signal_id, s.ticker_a, ...`). Ensure the JOIN uses `s.*`-style explicit selection, not `SELECT *`.

### Pitfall 6: The Reactivation Guard Requires Both Code Sites
**What goes wrong:** `add_pairs()` in `api/routes/pairs.py` and `add_pairs_batch()` in `ui/app.py` both handle reactivation (setting `is_active = 1`). Only fixing one means the guard is bypassable.
**Why it happens:** The same reactivation logic is duplicated in two places.
**How to avoid:** Both code sites must also set `reactivated_at = current_timestamp` when running the `UPDATE ticker_pairs SET is_active = 1` statement.
