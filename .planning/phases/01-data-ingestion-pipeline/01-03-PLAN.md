    **ui/app.py:**

    Function `create_app(config: AppConfig) -> gr.Blocks`:

    Create the PolygonClient and DB connection inside create_app using config values. Call init_schema on the connection.

    Build `gr.Blocks(title="Lead-Lag Quant")` with two tabs:

    **Tab 1: "Pair Management" (UI-06)**

    Layout:
    - Row with two Textbox inputs: "Leader Ticker" (placeholder "e.g., NVDA") and "Follower Ticker" (placeholder "e.g., CRWV")
    - Row with "Add Pair" button (variant="primary") and "Refresh" button
    - Textbox "Status" (interactive=False) for validation/save feedback
    - Dataframe "pair_table" showing active pairs (columns: ID, Leader, Follower, Created, Active)

    Callbacks:
    - `add_pair(leader: str, follower: str)`:
      1. Strip whitespace, convert to uppercase
      2. Validate both are non-empty, return error if empty
      3. Validate leader != follower, return error if same
      4. Call `client.get_ticker_details(leader)` -- if None, return error: f"Invalid ticker: {leader} (not found or inactive on Polygon.io)"
      5. Call `client.get_ticker_details(follower)` -- if None, return error for follower
      6. Insert into ticker_pairs: `INSERT INTO ticker_pairs (leader, follower) VALUES (?, ?)` -- catch IntegrityError for duplicate pairs
      7. Return success status message and refreshed pair table
    - `refresh_pairs()`:
      1. Query `SELECT id, leader, follower, created_at, is_active FROM ticker_pairs ORDER BY created_at DESC`
      2. Return as list of lists for Dataframe

    Wire: add_pair_btn.click(fn=add_pair, inputs=[leader_input, follower_input], outputs=[pair_status, pair_table])
    Wire: refresh_btn.click(fn=refresh_pairs, outputs=[pair_table])
    Load pairs on app start: app.load(fn=refresh_pairs, outputs=[pair_table])

    **Tab 2: "Data Ingestion" (INGEST-01)**

    Layout:
    - Textbox "from_date" (label="From Date", value="2020-01-01", placeholder="YYYY-MM-DD")
    - Textbox "to_date" (label="To Date", value=today's date as YYYY-MM-DD string, placeholder="YYYY-MM-DD")
    - Button "Fetch All Pairs" (variant="primary")
    - Textbox "fetch_log" (label="Ingestion Log", lines=15, interactive=False) for streaming output

    Callbacks:
    - `fetch_all_data(from_date: str, to_date: str, progress=gr.Progress())`:
      1. Validate date formats (YYYY-MM-DD)
      2. Query active pairs: `SELECT leader, follower FROM ticker_pairs WHERE is_active = 1`
      3. If no active pairs, return "No active pairs. Add pairs in the Pair Management tab first."
      4. For each pair, call `ingest_pair(client, conn, leader, follower, from_date, to_date)` from ingestion_massive.ingestion
      5. Update gr.Progress after each pair completes: `progress((i+1)/total, desc=f"Fetching {leader}/{follower}...")`
      6. Accumulate a log string showing per-pair results (ticker counts for aggs, splits, dividends)
      7. Return the complete log string
      8. Wrap in try/except: on error, return error message in the log

    Wire: fetch_btn.click(fn=fetch_all_data, inputs=[from_date, to_date], outputs=[fetch_log])

    CRITICAL: Call `app.queue()` before returning the app (required for gr.Progress to work).

    **main.py:**

    ```python
    """Lead-Lag Quant application entry point."""
    from utils.config import get_config
    from utils.logging import configure_logging, get_logger
    from ui.app import create_app

    def main():
        configure_logging()
        log = get_logger("main")
        config = get_config()
        log.info("starting_app", db_path=config.db_path, plan_tier=config.plan_tier.value)
        app = create_app(config)
        app.launch(server_name="0.0.0.0", server_port=7860)

    if __name__ == "__main__":
        main()
    ```

    The app should be launchable with: `cd C:/Users/USER/lead-lag-quant && uv run python main.py`

    - Do NOT use gr.State() for persistent data -- SQLite is the source of truth

    - All ticker inputs should be uppercased and stripped before processing
    - The Dataframe for pairs should refresh after every add_pair call
    - Error messages should be user-friendly (not raw tracebacks)
