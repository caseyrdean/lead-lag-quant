# Milestone v1.0: MVP

**Status:** ✅ SHIPPED 2026-03-21
**Phases:** 1–6.1 (8 phases, 2 decimal insertions)
**Total Plans:** 18

## Overview

Local quantitative analytics application that ingests unadjusted equity data from Polygon.io, applies deterministic split-adjustment (Policy A), computes rolling cross-correlation lead-lag signals for user-seeded ticker pairs, generates full position specs with strict confidence thresholds, validates signal quality through paper trading simulation and stored-data backtesting, and exposes a FastAPI backend with React/Vite frontend.

The six core phases follow the pipeline's strict dependency chain. Two decimal phases were inserted for critical fixes.

## Phases

### Phase 1: Data Ingestion Pipeline

**Goal**: User can add ticker pairs in the app and fetch complete, deduplicated raw market data from Polygon.io into SQLite
**Depends on**: Nothing (first phase)
**Plans**: 3 plans

Plans:
- [x] 01-01-PLAN.md -- Project scaffolding, SQLite schema (ticker_pairs, raw_api_responses, ingestion_log), AppConfig, structlog, NYSE calendar helpers, tests
- [x] 01-02-PLAN.md -- Polygon.io REST client (pagination via next_url, token-bucket rate limiter, 429 backoff) and ingestion module; SPY auto-included via set deduplication
- [x] 01-03-PLAN.md -- Gradio app shell with pair management panel (UI-06); SQLite as single source of truth; app.queue() for gr.Progress

**Details:**
- Used raw sqlite3 (no ORM) for full control over schema and ON CONFLICT clauses
- Module-level NYSE calendar caching to avoid expensive re-creation
- Always pass adjusted=false to Polygon /v2/aggs for unadjusted raw prices
- Deterministic params serialization via json.dumps(sort_keys=True) for idempotent row lookup

### Phase 2: Normalization & Returns

**Goal**: Raw ingested data is transformed into split-adjusted bars and policy-tagged return series that all downstream computation can trust
**Depends on**: Phase 1
**Plans**: 2 plans

Plans:
- [x] 02-01-PLAN.md -- Schema extension (4 new tables), normalization module (split adjuster, bar normalizer, dividend storer, timestamp utils, orchestrator), tests
- [x] 02-02-PLAN.md -- Returns computation (multi-period pct_change from adj_close), Normalize tab in Gradio UI

**Details:**
- Policy A is split-only: adj_price = raw_price * historical_adjustment_factor; dividends stored separately and never applied to prices
- XNYS calendar instantiated once as module-level singleton in timestamp_utils.py
- fetched_at on splits rows copied from raw_api_responses.retrieved_at for point-in-time backtest isolation (NORM-05)
- Returns computed strictly per-ticker using fill_method=None for pandas >= 2.1 compatibility

### Phase 3: Feature Engineering

**Goal**: Normalized return series are transformed into statistically rigorous features: residualized cross-correlations, relative strength, volatility, and standardized metrics
**Depends on**: Phase 2
**Plans**: 2 plans

Plans:
- [x] 03-01-PLAN.md -- Add scipy/statsmodels, extend SQLite schema with 5 feature tables, SPY residualization and rolling cross-correlation with Bonferroni (FEAT-01, FEAT-02, FEAT-03)
- [x] 03-02-PLAN.md -- Relative strength, volatility, z-scores, lagged returns, and pipeline orchestrator (FEAT-04, FEAT-05, FEAT-06, FEAT-07)

**Details:**
- BONFERRONI_THRESHOLD = 0.05/11 — 11 lag tests per window requires correction to avoid ~42% false positive rate
- statsmodels 0.14 RollingOLS: residuals computed manually as y - (alpha + beta*spy) from rolling params DataFrame
- Manual Python loop for rolling cross-correlation — pandas.rolling().apply() is 1D only
- Pipeline separates pair-level features (xcorr, RS) from per-ticker features (volatility, zscore, lagged_returns)

### Phase 4: Lead-Lag Engine, Regime & Signals

**Goal**: Features are consumed to detect statistically stable lead-lag relationships, classify market regime, and generate full position specs that meet strict confidence thresholds
**Depends on**: Phase 3
**Plans**: 2 plans

Plans:
- [x] 04-01-PLAN.md -- leadlag_engine package: SQLite schema (4 new tables), optimal lag detector (ENGINE-01), RSI-v2 stability scorer (ENGINE-02) with tests
- [x] 04-02-PLAN.md -- Regime classifier (REGIME-01), distribution detector (REGIME-02), signal generator with hard gate (ENGINE-03, SIGNAL-01/02), pipeline orchestrator

**Details:**
- RSI-v2 weights: lag_persistence=0.30, walk_forward_oos=0.25, rolling_confirmation=0.20, regime_stability=0.15, lag_drift=0.10
- detect_optimal_lag requires MIN_SIGNIFICANT_DAYS=30 per lag; selects by abs(median_corr)
- passes_gate uses abs(correlation_strength) > 0.65 — allows short signals (strong inverse correlations)
- classify_regime() MUST be called before compute_stability_score() — ordering enforced in pipeline.py

## Milestone Summary

**Decimal Phases:**
- Phase 5.1: API Security & Data Integrity Fixes (inserted after Phase 5 for critical bugfixes uncovered during frontend integration)
- Phase 6.1: Signal Gate Threshold Fix (inserted after Phase 6 to close the last open gap from milestone audit — ENGINE-03 constants were shipped at wrong values)

**Key Decisions:**
- v1 is a local FastAPI + React app — no AWS Lambda, S3, DynamoDB, or Terraform. SQLite for all storage.
- Gradio UI fully removed in Phase 5.1-04; FastAPI backend + React/Vite frontend is the production stack
- Policy A (split-only adjustment) is canonical — dividends stored separately, never applied to prices
- Strict signal thresholds: stability_score > 70, correlation_strength > 0.65; no exceptions, no overrides
- Average-cost basis for paper trading (not FIFO)
- reload=False in uvicorn.run — incompatible with background threads (PipelineScheduler, BackgroundPricePoller)

**Issues Resolved:**
- ENGINE-03 threshold mismatch: constants shipped at 50.0/0.50 instead of 70.0/0.65 (closed in Phase 6.1)
- WebSocket broadcast unreliable from background threads — fixed with run_coroutine_threadsafe
- Concurrent auto-execute double-spend risk — fixed with threading.Lock
- Soft-deleted pairs appearing in signal queries — fixed with INNER JOIN is_active=1 pattern

**Issues Deferred:**
- 2 pre-existing test failures in test_engine_detector.py (test_detect_optimal_lag_insufficient_days, test_detect_optimal_lag_ignores_null_correlation) — documented, not blocking

**Technical Debt Incurred:**
- Gradio UI panels for Phases 1–5 were built and then discarded when FastAPI/React became the production stack in Phase 5.1
