## v1 Requirements — Final Status

### Data Ingestion

- [x] **INGEST-01**: User can enter a ticker pair in the Gradio UI and trigger a data fetch from Polygon.io
- [x] **INGEST-02**: System fetches unadjusted aggregate bars (`adjusted=false`) for all tickers in active pairs
- [x] **INGEST-03**: System fetches split records via `/v3/reference/splits` for each ticker
- [x] **INGEST-04**: System fetches dividend records via `/v3/reference/dividends` for each ticker (stored, never applied to returns)
- [x] **INGEST-05**: All raw Polygon API responses stored to SQLite as JSON system of record with `retrieved_at`, `ticker`, `endpoint`, `request_params`
- [x] **INGEST-06**: Re-running ingestion for an existing date range does not create duplicate records (idempotent)
- [x] **INGEST-07**: Polygon client handles cursor-based pagination via `next_url` until exhausted
- [x] **INGEST-08**: Polygon client implements token-bucket rate limiter configurable by plan tier
- [x] **INGEST-09**: Polygon client retries on HTTP 429 with exponential backoff and jitter (max 5 retries)
- [x] **INGEST-10**: System also fetches SPY aggregate bars automatically (required for residualization)

### Normalization

- [x] **NORM-01**: System applies Adjustment Policy A: split-adjust OHLCV prices using cumulative split ratio from fetched split records
- [x] **NORM-02**: Dividends are stored in their own table and never applied to price calculations under Policy A
- [x] **NORM-03**: `adjustment_policy_id` field (value: `"policy_a"`) is stored on every normalized bar, return, and signal record
- [x] **NORM-04**: `returns_policy_a` computed from `adj_close` exclusively (1d, 5d, 10d, 20d, 60d rolling)
- [x] **NORM-05**: Split records stored with `fetched_at` timestamp to enable point-in-time backtest isolation
- [x] **NORM-06**: Timestamps normalized from Polygon Unix milliseconds to UTC datetime; trading day assigned per NYSE calendar

### Feature Engineering

- [x] **FEAT-01**: Rolling cross-correlation computed across lags -5 to +5 bars using `scipy.signal.correlate` on a minimum 60-day rolling window
- [x] **FEAT-02**: Returns for both tickers residualized against SPY returns before cross-correlation computation
- [x] **FEAT-03**: Bonferroni correction applied to significance testing across 11 lag offsets (threshold = 0.05/11 ≈ 0.0045)
- [x] **FEAT-04**: Relative Strength (RS) computed as cumulative_return(leader, 10d) - cumulative_return(follower, 10d), rolling
- [x] **FEAT-05**: Rolling volatility computed on 20-day window
- [x] **FEAT-06**: Z-score standardized returns computed per ticker
- [x] **FEAT-07**: Lagged returns computed at offsets ±1 through ±5 bars

### Lead-Lag Engine

- [x] **ENGINE-01**: Optimal lag detected per pair as the lag offset with maximum stable cross-correlation
- [x] **ENGINE-02**: RSI-v2 stability score computed as composite of: lag persistence consistency + regime stability + rolling window confirmation + walk-forward OOS validation (non-overlapping estimation/gap/validation windows) + lag drift penalty → scalar 0–100
- [x] **ENGINE-03**: Hard threshold gate: signals only generated when stability_score > 70 AND correlation_strength > 0.65 — *closed in Phase 6.1; constants were shipped at 50.0/0.50, corrected to 70.0/0.65*
- [x] **ENGINE-04**: Signals stored immutably in SQLite with full explainability payload: optimal_lag, window_length, correlation_strength, stability_score, regime_state, adjustment_policy_id, generated_at

### Regime & Distribution

- [x] **REGIME-01**: Regime classified as Bull / Base / Bear / Failure using hard rules: MA structure (price vs 21d and 50d MA), RS thresholds (Bull: RS > +5% for 10 sessions; Bear: RS < -7% for 5 sessions), ATR regime (expansion > 130% of 20d avg; compression < 80%)
- [x] **REGIME-02**: Distribution detection: flags down days with volume > 150% of 30-day average AND VWAP rejection streaks ≥ 3 consecutive sessions

### Signal Generation

- [x] **SIGNAL-01**: System generates full position spec for each qualifying signal: entry condition (date + direction), expected target (historical mean return during lag window), invalidation rule (leader reversal % threshold), sizing tier (full/half/quarter based on stability_score)
- [x] **SIGNAL-02**: System generates directed flow map entry (A leads B notation) for each active pair

### Paper Trading Simulation

- [x] **TRADE-01**: User can set starting paper capital amount in the app
- [x] **TRADE-02**: System auto-executes simulated trades when a qualifying signal fires, sizing position per the signal's sizing tier
- [x] **TRADE-03**: User can manually enter a Buy or Sell for any ticker with a custom share quantity at any time via the trading UI
- [x] **TRADE-04**: Open positions display entry price, current price (15-min delayed), and unrealized P&L per position, updated automatically
- [x] **TRADE-05**: System polls Polygon snapshot endpoint every 15 minutes during market hours (9:30–16:00 ET) to refresh current prices
- [x] **TRADE-06**: Realized P&L recorded and stored in SQLite when a position is closed
- [x] **TRADE-07**: Position flagged for exit in UI when leader reversal exceeds the signal's invalidation threshold
- [x] **TRADE-08**: Trade history (all opens and closes) stored in SQLite with timestamps and P&L

### Bug Fixes (Phase 5.1)

- [x] **BUGFIX-01**: Free-tier pair limit enforced server-side in FastAPI (not only Gradio)
- [x] **BUGFIX-02**: Pair soft-delete via FastAPI DELETE endpoint commits to SQLite atomically
- [x] **BUGFIX-03**: Signals never include signals for soft-deleted (is_active=0) pairs
- [x] **BUGFIX-04**: Signals from a deactivation period not auto-executed when pair reactivated
- [x] **BUGFIX-05**: Concurrent buy/sell operations serialized via threading.Lock
- [x] **BUGFIX-06**: POST /api/trading/buy and /api/trading/sell reject invalid inputs with 422
- [x] **BUGFIX-07**: All analytics endpoints return structured JSON error response on exception
- [x] **BUGFIX-08**: broadcast_sync() delivers WebSocket messages reliably via run_coroutine_threadsafe

### Backtest

- [x] **BACKTEST-01**: Backtest module reads from SQLite stored data only — never calls Polygon API
- [x] **BACKTEST-02**: Backtest adjustment path filters split records to `fetched_at <= backtest_date` (no look-ahead bias)
- [x] **BACKTEST-03**: Backtest reports: hit rate, mean return per trade, annualized Sharpe ratio, maximum drawdown

### UI

- [x] **UI-01**: Signal Dashboard panel — displays active signals with full position spec; includes auto-execute toggle for paper trading
- [x] **UI-02**: Lead-Lag Charts panel — cross-correlation heatmap across lags, rolling optimal correlation over time for selected pair
- [x] **UI-03**: Regime State panel — current Bull/Base/Bear/Failure classification with all indicator values (RS, MA position, ATR, volume)
- [x] **UI-04**: Paper Trading panel — open positions table, total portfolio P&L, win rate, manual Buy/Sell inputs, closed trade history
- [x] **UI-05**: Backtest Results panel — hit rate, mean return, Sharpe, max drawdown for user-selected pair and date range
- [x] **UI-06**: User can add a new ticker pair (leader + follower) via text inputs in the app; system validates tickers against Polygon before saving
