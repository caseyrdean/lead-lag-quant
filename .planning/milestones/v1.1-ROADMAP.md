# Milestone v1.1: Outperformance Signal Enhancement

**Status:** ✅ SHIPPED 2026-03-21
**Phases:** 7

## Overview

Extends the signal generator beyond binary lead-lag correlation detection to indicate whether the follower is actively outpacing the leader. Adds BUY/HOLD/SELL action classification, RS acceleration, response window, and outperformance margin to every signal record. Updates the backtest engine to disaggregate metrics by action. Pipeline scheduler reduced from 30-minute to 15-minute polling.

## Milestone Summary

**Key Decisions:**
- SELL priority before BUY condition 1 — positive-but-declining RS is a warning signal, not a buy; BUY only fires on non-declining positive RS
- classify_action is a pure function (no DB calls) — takes pre-computed rs_series, rs_std, rs_mean for testability
- All 5 new signal columns nullable — sparse-history pairs must not crash on insert
- signal_transitions only logs rows on state change (BUY→HOLD, HOLD→SELL) — no HOLD→HOLD spam
- by_action always returns all 4 keys (BUY/HOLD/SELL/UNKNOWN) with zero-dicts for empty groups — prevents future KeyError in UI code
- outperformance_vs_leader falls back to 0.0 (not None) when no leader return data available
- leader_rs_deceleration computed inline in generate_signal (one-off 10-row polyfit, not a separate helper)
- response_window returns None until ≥2 complete BUY→exit cycles — bootstrap behavior, not a bug

**Issues Resolved:**
- Plan checker blocker: `<reuse existing sharpe formula>` placeholder tokens replaced with actual numpy formulas before execution
- Executor auto-fixed SELL priority ordering vs plan text mismatch — plan text listed BUY first but test contract asserted SELL for declining RS

**Technical Debt Incurred:**
- React UI for BUY/HOLD/SELL deferred — no dashboard changes in v1.1; action field is backend-only until v1.2+
- response_window will return None for all pairs until sufficient BUY→exit history accumulates in signal_transitions
